Ananse’s Web Protocol

The Concept: Instead of trusting a Google server with your keys, you trust your "Village." Ananse’s Web uses Shamir's Secret Sharing (SSS) to split your master key into shards distributed among trusted family members (Guardians).
The Mechanism
  • Shard Generation: Master Key is split into 5 shards (2 kept on device, 3 sent to Guardians).
  • Reconstruction: Requires 3 shards to rebuild the key.
  • Zero-Knowledge: Guardians never see the key, only their encrypted shard.
  • Social Recovery: Lose your phone? Guardians approve a "Help" request to restore your vault.
Transaction Signing
// Rust Psuedocode for Threshold Sign
fn approve_transaction(tx: Transaction) {
  if tx.amount > 500.0 {
    // Require Witness
    let witness_sig = Ananse::request_witness(guardian_id);
    let local_sig = Vault::sign(tx);
    Chain::submit(tx, [local_sig, witness_sig]);
  }
}

Technical Stack (v2.1)

Architecture Overview

Qyra is built primarily in Rust to ensure memory safety on low-resource devices. The stack is divided into the Client Layer (UI/Interaction) and the Compute Layer (Brain/Relay).

1. Frontend (Client)
Tech/Crate Role
Leptos Web/WASM Reactive UI framework for the App interface.
Tauri v2 Desktop wrapper for the "Desktop Satellite" binary (Windows/Mac).
UniFFI Mobile Rust bindings for Kotlin (Android) & Swift (iOS).
DaisyUI Styles Tailwind components for the Kente UI theme.
2. Backend (Home Node / Relay)
Tech/Crate Role
Axum Server High-performance HTTP server for the Cloud Relay.
SurrealDB Database Stores User Profile, Transaction Logs, and Graph relations.
Tokio Runtime Async runtime for handling concurrent tasks.
Docker Ops Containerization for easy Home Node deployment (Pi).
3. AI Intelligence
Tech/Crate Role
Candle-Core Inference Runs quantized LLMs (Phi-3) directly in Rust.
Whisper-RS Voice C++ bindings for offline speech-to-text (Twi/Yoruba).
LanceDB Memory Vector store for semantic search and long-term recall.
4. Ananse Security & Net
Tech/Crate Role
SSSS Crypto Shamir's Secret Sharing for key fragmentation.
Threshold_Crypto Signatures BLS threshold signatures for multi-guardian approval.
LibP2P Network Peer-to-peer tunnels (NAT Traversal) for Desktop Link.
AES-GCM Vault Local encryption for the shard stored on-device.

Flow: Ananse Recovery
  1. Initiation: User installs fresh app, taps "Recover Account".
  2. Discovery: User enters their public username (`@kofi`).
  3. Challenge: App contacts the Guardian Registry (Smart Contract/DHT) to find Guardians.
  4. Request: App sends a push notification to Guardians: "Kofi needs help."
  5. Approval: Guardian taps "Approve" (Signs a partial key shard).
  6. Reconstruction: Once 3/5 Guardians approve, the App reconstructs the Private Key in memory.
Flow: Desktop Command
  1. Input: User speaks "Open the budget file on my laptop."
  2. Intent: Mobile NPU detects intent `desktop_file_open`.
  3. Routing: App checks `linked_device` table for online Satellite.
  4. Transport: Message sent via encrypted P2P tunnel (or Relay fallback).
  5. Execution: Laptop Satellite receives JSON, validates signature, executes OS command.
  6. Feedback: Laptop sends success status back to Phone.
QA & Testing Strategy
  • Voice Validation: Use "Common Voice" datasets + Community Beta program in Accra/Lagos for accent verification.
  • Network Simulation: Test P2P protocol under "2G/Edge" network conditions using `toxiproxy`.
  • Security Audit: External audit of the Rust `ssss` crate implementation before v1.0 release.

Risk & Feasibility Analysis

Cost Feasibility

Risk: Hosting the "Tier 1" Cloud Relay costs money (GPU servers).
Mitigation: Freemium model. The "Cloud" tier has rate limits. "Local" tier (Home Node) is unlimited because the user pays for their own hardware.

Cultural Feasibility

Status: High

The Ananse Protocol fits the "High Trust" communal culture of West Africa perfectly. Users are more likely to trust a brother than a cloud server.
